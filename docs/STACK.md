# Technology Stack

`pkgto-web` is a GTO preflop equity analyzer that runs entirely in the browser with no server-side compute. The Rust poker engine is compiled to WebAssembly and called directly from a vanilla JavaScript front-end.

---

## Core Language — Rust

All computation logic is written in [Rust](https://www.rust-lang.org/), a systems language focused on memory safety and performance without a garbage collector. The crate targets `wasm32-unknown-unknown`, meaning it compiles to a pure WebAssembly binary with no operating-system dependencies.

- **Minimum supported version:** 1.91.0 (set via `rust-version` in `Cargo.toml`)
- **Edition:** 2024
- **Learn more:**
  - [The Rust Book](https://doc.rust-lang.org/book/)
  - [Rust by Example](https://doc.rust-lang.org/rust-by-example/)
  - [Rustlings (hands-on exercises)](https://github.com/rust-lang/rustlings)

## Poker Engine — pkcore

[`pkcore`](https://crates.io/crates/pkcore) is the domain library that provides all poker primitives:

| Type | Purpose |
|------|---------|
| `Two` | Parses and represents a two-card hole hand |
| `Combos` | Parses a range string (e.g. `66+,AJs+,KQo`) into a set of hand combos |
| `Versus` | Runs preflop all-in equity calculations between a hand and a range |
| `HUPResult` | Holds win/loss/draw counts for a single heads-up pairing |

The library handles card parsing, blocker removal, combo enumeration, and enumeration-based equity computation. `pkgto-web` is essentially a thin WebAssembly wrapper around `pkcore`.

## WebAssembly (WASM)

[WebAssembly](https://webassembly.org/) is a binary instruction format that browsers execute at near-native speed. The Rust crate is compiled to a `.wasm` binary which is loaded by the browser at startup via the ES module entry point generated by `wasm-pack`.

- **Target triple:** `wasm32-unknown-unknown`
- **Learn more:**
  - [WebAssembly official site](https://webassembly.org/)
  - [MDN: WebAssembly](https://developer.mozilla.org/en-US/docs/WebAssembly)
  - [Rust and WebAssembly book](https://rustwasm.github.io/docs/book/)

## wasm-bindgen

[`wasm-bindgen`](https://github.com/rustwasm/wasm-bindgen) generates the JavaScript/TypeScript glue code that bridges the WASM binary and the browser. It handles type conversions (Rust `&str` ↔ JS `string`, etc.) and exposes Rust functions as ordinary ES module exports.

Annotating a Rust function with `#[wasm_bindgen]` causes `wasm-pack` to generate a corresponding typed JS wrapper. The single exported function is:

```rust
#[wasm_bindgen]
pub fn analyze_gto(hero: &str, villain_range: &str) -> String { … }
```

- **Learn more:**
  - [wasm-bindgen guide](https://rustwasm.github.io/docs/wasm-bindgen/)
  - [wasm-bindgen API docs](https://docs.rs/wasm-bindgen)

## wasm-pack

[`wasm-pack`](https://rustwasm.github.io/wasm-pack/) is the build tool that orchestrates the full compilation pipeline:

1. Runs `cargo build --target wasm32-unknown-unknown`
2. Runs `wasm-bindgen` to produce JS/TS bindings
3. Optionally runs `wasm-opt` to shrink and optimise the binary
4. Writes the output (`pkgto_web.js`, `pkgto_web.d.ts`, `pkgto_web_bg.wasm`) into `www/pkg/`

Two build profiles are used:

| Command | Profile | Use |
|---------|---------|-----|
| `make build` | debug | Local development |
| `make build-release` | release + `wasm-opt` | Production / CI |

- **Learn more:**
  - [wasm-pack book](https://rustwasm.github.io/docs/wasm-pack/)

## serde / serde_json

[`serde`](https://serde.rs/) is Rust's de-facto serialization framework. Structs annotated with `#[derive(Serialize)]` can be turned into JSON with a single call to `serde_json::to_string(…)`.

`analyze_gto` serializes its result (or error) to a JSON string and returns it to JavaScript, where `JSON.parse()` deserializes it. This is the simplest cross-WASM-boundary data protocol — no custom binary format required.

- **Learn more:**
  - [serde.rs](https://serde.rs/)
  - [serde_json docs](https://docs.rs/serde_json)

## console_error_panic_hook

[`console_error_panic_hook`](https://github.com/rustwasm/console_error_panic_hook) redirects Rust panics to the browser's `console.error`, making debugging significantly easier. It is installed once at module startup via the `#[wasm_bindgen(start)]` entry point.

Without this hook, a panic produces an opaque `RuntimeError: unreachable executed` in the browser console. With it, the full Rust panic message and backtrace appear.

## getrandom (WASM target only)

[`getrandom`](https://docs.rs/getrandom) provides a cross-platform random-number source. Under the `wasm32-unknown-unknown` target it must be explicitly told to use the browser's `crypto.getRandomValues` API via the `js` feature flag:

```toml
[target.'cfg(target_arch = "wasm32")'.dependencies]
getrandom = { version = "0.2", features = ["js"] }
```

This dependency is pulled in transitively by `pkcore` and does not surface in application code directly.

- **Learn more:**
  - [getrandom docs](https://docs.rs/getrandom)

## Front-end — Vanilla HTML/CSS/JS

The UI (`www/index.html`) is a single self-contained HTML file with no framework, bundler, or build step. Key techniques used:

| Feature | How it's done |
|---------|--------------|
| WASM loading | ES module `import init, { analyze_gto } from './pkg/pkgto_web.js'` |
| Layout | CSS Grid (`grid-template-columns`) for the form and stats panels |
| Responsive design | `@media (max-width: 600px)` breakpoint collapses the two-column form to one |
| Range chart | 13×13 CSS Grid visualising pairs / suited / offsuit hands |
| Shareable URLs | `URLSearchParams` encodes the current hero hand and villain range into the URL |
| Loading indicator | CSS `@keyframes` spinner shown while the WASM call runs inside a `setTimeout` (yields to the event loop so the spinner renders first) |

## CI/CD — GitHub Actions + GitHub Pages

Deployment is fully automated via `.github/workflows/deploy.yml`:

1. **Checkout** — `actions/checkout@v4`
2. **Rust toolchain** — `dtolnay/rust-toolchain@stable` with the `wasm32-unknown-unknown` target
3. **Cargo cache** — `actions/cache@v4` keyed on `Cargo.lock` to speed up repeated builds
4. **wasm-pack install** — downloaded via the official installer script
5. **Release WASM build** — `wasm-pack build --release --target web --out-dir www/pkg`
6. **Pages upload** — `actions/upload-pages-artifact@v3` uploads the `www/` directory
7. **Deploy** — `actions/deploy-pages@v4` publishes to GitHub Pages

The workflow triggers on every push to `main` and can also be triggered manually (`workflow_dispatch`). A `concurrency` guard ensures that a second push does not cancel an in-flight deployment.

- **Learn more:**
  - [GitHub Actions docs](https://docs.github.com/en/actions)
  - [GitHub Pages docs](https://docs.github.com/en/pages)
  - [dtolnay/rust-toolchain action](https://github.com/dtolnay/rust-toolchain)

## Local Development

```bash
# Install Rust (if needed)
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

# Add the WASM compile target
rustup target add wasm32-unknown-unknown

# Install wasm-pack
curl https://rustwasm.github.io/wasm-pack/installer/init.sh -sSf | sh

# Build and serve on http://localhost:8080
make serve
```
